## Phase 3b â€” sanity checks

`scripts/audit_sanity_checks.py`. Eight invariants per strategy:

| Check | Description |
|---|---|
| C1 | `filter_marginals.passed_in_combination â‰ˇ count(entry_audit.all_clauses_pass)` |
| C2 | `entry_audit.clause_scanner_pass â‰¤ scanner artifact pass count` |
| C3 | `trade_log_audit > simulator_trade_log` (capacity-constraint) |
| C4 | every simulator entry has a matching audit entry |
| C5 | `simulator.exit_reason âŠ† audit.exit_reason` |
| C6 | both IS and OOS populated in entry_audit + trade_log_audit |
| C7 | no NULL period rows |
| C8 | `all_clauses_pass â‰ˇ AND(clause_*)` |

**Both champions pass all 8 checks.**

Two findings surfaced:

1. **eod_b's simulator exit_reason is uniformly `"natural"`** (1,795
   of 1,795 trades). The signal generator does not populate
   `entry_order["exit_reason"]`, so the simulator's
   `exit_order.get("exit_reason", "natural")` fallback fires every
   time. The audit's exit_reason (from HOOK 3a) is the authoritative
   source for eod_b. C5 raises a soft warning explaining this.

2. **C2 had a sub-1% gap between scanner_snapshot and entry_audit
   (2,454 rows)**. Diagnosed as warm-up/boundary trim â€” rows that
   appear in scanner_snapshot but get dropped before the entry-clause
   stage runs. Where rows appear in BOTH artifacts, `scanner_pass`
   and `clause_scanner_pass` agree 100%. Adjusted C2 to be a subset
   check (`audit â‰¤ scanner`), now passes.

3. **Sub-pence price diffs**: 119 of 1,795 audit-vs-sim entry_prices
   differed by < 1e-3 (e.g. `200.39604` audit vs `200.3960`
   simulator). The simulator stores 4-decimal precision; the audit
   keeps full float. Tolerance relaxed to 1e-3; 100% match on both
   strategies.
